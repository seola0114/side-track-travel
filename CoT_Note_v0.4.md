# CoT (Chain of Thought) Note v0.4

## 1. Abduction — 문제 발견

AI를 이용해서 여행 계획을 짤 때의 불편함은 무엇일까? 실제 내 사례를 생각해보면 난 계획표를 AI에게 시켜서 HTML로 만들었다. AI에서 완성된 계획표를 보기가 힘들었기 때문이다. 다른 사람들은 계획표를 어떤 포맷으로 가지고 가는지 궁금해서 찾아보니 예전에 책으로 만들어서 들고다녔다는 사람은 웹 형태로 만들고, 4살 딸이 있는 30대 이상 부모의 경우는 PDF로 다운로드 후 인쇄하는 형태를 채택하였다. 결론적으로 AI 내에서 결과물이 보기 힘들다는것. 그럼 이 문제를 어떻게 해결하면 좋을까?

### 핵심마찰 고민
1. 클로드에서 여행계획을 볼 수 없어서 그걸로 HTML, PDF, 출력물을 만들어서 가지고 다닌다.
2. AI에만 의존해서 계획하고 다니고 있었는데 갑자기 여행계획이 바뀌어서 AI가 짰던 여행계획을 변경해야한다
→ 여행 계획은 고정과 유동을 동시에 요구하는데, AI 챗은 둘 중 하나만 준다.

### 핵심 질문
- 다른 페이지로 이동했다가 해당 페이지로 다시 돌아가려면 어떻게 해야 하는가?
- 일정에 변수가 생겼을 때, AI가 변경할 수 있는 부분은 자동으로 변경하고, 예약 변경 등 추가로 사용자의 행동이 필요한 항목을 쉽게 알고 편하게 재계획 할 수 있도록 한다.

## 2. AI — 사용자 목표 정의

AI가 짠 일정에 변수가 생겼을 때, 일정을 처음부터 다시 짜거나 조건을 다시 설명하지 않고, 영향받는 부분만 골라 고쳐 다음 행동을 바로 정한다.

**완료 조건**: 변수를 입력한 뒤 몇 분 안에, 무엇이 그대로 유지되고 무엇을 바꿔야 하는지가 분간된 채로 일정이 다시 확정된다.

## 3. Gap Analysis — As-Is vs To-Be

### As-Is (기존 클로드 앱)
- 일정이 텍스트 로그로만 존재
- 대화가 길어지면 이전 일정이 스크롤 위로 밀림
- 변수 발생 시 → AI가 전체 일정을 텍스트로 다시 출력
- 사용자가 직접 이전 출력과 비교해서 뭐가 바뀌었는지 찾아야 함
- A/B 비교 없이 텍스트 설명만 제공

### To-Be (프로토타입)
- 일정표가 대화와 별도로 항상 보임 (상단 접기/펼치기)
- 변수 입력 → 영향받는 셀만 시각적으로 표시 (broken, changed)
- A/B안을 모달에서 나란히 비교, 추천 안에 이유 표시
- 한 번 탭으로 선택 → 일정표 즉시 반영 (flash + 자동 스크롤)
- 마음 바뀌면 "다른 안으로 변경"으로 재선택 가능

### 핵심 차이
| | As-Is | To-Be |
|---|---|---|
| 일정 형태 | 텍스트 | 시각 일정표 |
| 변수 대응 | 전체 재출력 | 영향 부분만 표시 |
| 선택 방식 | 텍스트 설명 읽기 | A/B 비교 모달 |
| 반영 확인 | 직접 스크롤해서 찾기 | 셀 flash + 자동 스크롤 |
| 되돌리기 | 불가 | 재선택 가능 |

## 4. Takeaway — 4주차 의사결정 기록

### 4-1. 3주차 → 4주차 변경 판단
- 3주차 프로토타입을 베이스로 유지하고, 새로 만들지 않기로 결정
- **이유**: 3주차에서 이미 일정표+대화 split 레이아웃, 사이드바, composer 등 완성도 높은 UI가 구축됨. 새로 만들면 품질이 떨어지고 시간 낭비.

### 4-2. "키스크린 바로 보기" 진입점 추가
- 리뷰어가 풀 플로우(초안→교통→통신→지연)를 다 따라가지 않아도 핵심 순간에 바로 도달할 수 있도록 랜딩에 단축 버튼 추가
- **이유**: 과제 완성 기준이 "보자마자 어떤 불편이 해결되는지 알 수 있는 것"이므로, 키스크린까지의 거리를 최소화해야 함

### 4-3. delayReply 텍스트 압축
- 12줄 → 6줄로 줄임. "즉시 조치 / 자동 조정 / 영향 없음"을 각 1줄로.
- **이유**: 사용자가 공항에서 급하게 읽는 상황. "비교하기" 버튼이 스크롤 없이 바로 보여야 목표 달성 효율이 높음

### 4-4. A안 추천 표시 + 이유
- A안(카페 취소)에 추천 뱃지와 "3시간 지연 후 바로 카페로 가면 체력 소모가 큼" 이유 추가
- **이유**: 사용자가 낯선 환경에서 빠르게 결정해야 하는 상황. AI가 왜 이 안을 추천하는지 이유를 보여주면 결정 부담이 줄어듦

### 4-5. 재선택 가능
- A안 선택 후에도 "다른 안으로 변경" 버튼으로 B안으로 바꿀 수 있게
- **이유**: 되돌릴 수 없는 선택은 사용자에게 부담. "마음이 바뀌면 다시 조정할 수 있어요"로 안심감 제공

### 4-6. 보내기 시 일정표 자동 접기 / 변수 시 자동 펼치기
- 메시지 전송 → 일정표 접혀서 대화 공간 확보
- 지연 등 변수 입력 → 일정표 자동 펼쳐서 변경 사항 즉시 확인
- **이유**: 모바일 화면이 좁으므로, 상황에 따라 일정표/대화 중 더 중요한 쪽에 공간을 할당

### 4-7. 빼기 원칙 적용
- 일정표 "일정표" 타이틀 삭제 (자명한 라벨)
- 모달에서 "변경 전" 열 삭제 (A/B만으로 비교 충분)
- 모달 예약 충돌 설명 박스 삭제 (대화에서 이미 설명됨)
- 루틴 셀(출발/취침) 시각적 비중 축소
- 빈 셀 배경 제거
- **이유**: 과제 지침 "더 만들기보다 빼기"에 따라, 핵심 순간에 집중하도록 불필요한 정보를 제거
